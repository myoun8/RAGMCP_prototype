# rag-knowledge-pack-template

A RAG knowledge-pack template for NIST NCNR neutron-scattering instrument documentation, plus an agent/serving layer that answers questions over the packs and drives NCNR reduction tooling. Six packs under `rag/context_database/`: `candor/`, `vsans/`, `nse/`, `magik/`, `bt7/` (instrument-specific) and `common/` (shared NICE/NCNR-wide docs).

The repo has two layers, in two script directories:
- **Ingestion & evaluation** (`rag/scripts/`) — builds the RAG knowledge base: normalize → chunk → validate → embed, plus retrieval-quality evaluation. Packs and the Chroma store both live under `rag/`.
- **Agent & serving** (`scripts/`) — an MCP tool server plus two agent front-ends (web UI and CLI REPL) that retrieve from the knowledge base and call the reductus reduction backend + NCNR metadata API.

## Pack structure (per `PACK_STRUCTURE.md`)

Each pack (`rag/context_database/<pack>/`) contains:
- `README.md`, `source_inventory.csv`, `manifest.jsonl`, `glossary.yaml`, `access_policy.yaml`, `ingest_config.yaml`
- `originals/` — unmodified source material (web_pages, pdfs, papers, manuals, notebooks, scripts, data_examples)
- `normalized/` — Markdown docs with YAML frontmatter (doc_id, source_id, instrument, workflow_stage, source_type, access_level, status, owner, last_reviewed, source_url_or_path, citation_required), organized into stage subfolders (overview, instrument_control, experiment_planning, data_access/raw_data, metadata, sample_environment, troubleshooting, citations_publications, etc.)
- `chunks/` — JSONL chunk records (chunk_id, doc_id, source_id, text, metadata) generated from `normalized/`
- `eval/` — JSONL evaluation questions for retrieval testing
- `review/` — review artifacts

Schemas for these record types live in `schemas/`.

## Environment / keys

- `RCHAT_API_KEY` is read from a `.env` file in the repo root (or the environment) and is required by the ingestion normalization step and by `agent.py`. RChat is the NIST-hosted OpenAI-compatible proxy at `https://rchat.nist.gov/api/v1` fronting several internally-hosted models. Normalization uses the `openai` SDK against this endpoint (no Groq).
- `app.py` (the web UI) is bring-your-own-key: each browser session supplies its own API key(s) per provider (OpenAI / Anthropic / Google / RChat) in a UI modal, and a fresh agent is built per request from that key + model. Server-side default keys (`OPENAI_API_KEY`, `ANTHROPIC_API_KEY`, `GOOGLE_API_KEY`, `RCHAT_API_KEY`) are an optional fallback so it works out of the box.

## Ingestion & evaluation scripts (`rag/scripts/`)

The embedding/vectorstore layer is built on LangChain (`langchain-core`, `langchain-ollama`, `langchain-chroma`, `langchain-openai`). All embedding-dependent scripts require Ollama running locally with `nomic-embed-text` pulled (`ollama pull nomic-embed-text`); `_common.ensure_ollama()` auto-starts `ollama serve` and pulls the model if missing. The shared Chroma store lives at `rag/chroma_db/` (collection `ncnr_rag`).

- `run_pipeline.py` — **main entry point** that chains all four ingestion steps in order: normalize → chunk → validate → embed. Reads `RCHAT_API_KEY` from `.env` or the environment; exits early with a clear message if it is missing. Run as `python rag/scripts/run_pipeline.py [--pack PACK] [--model NAME] [--skip-normalize] [--skip-validate] [--dry-run]`. `--model` defaults to `gemma-4-31B-it`; `--pack` scopes normalization/chunking/validation to one pack (embed always ingests all available chunk files).
- `full_document_ingestion.py` — converts files in each pack's `originals/` to normalized Markdown using the RChat API (via the `openai` SDK against the RChat OpenAI-compatible endpoint). Skips files that already have a corresponding normalized `.md`. For each new file, streams the model's output then asks the user to confirm the target workflow stage before writing. Run as `python rag/scripts/full_document_ingestion.py --model NAME [--api-key KEY] [--pack PACK] [--dry-run]`. `--api-key` falls back to `RCHAT_API_KEY` env var. PDF support requires `pypdf` (`pip install pypdf`).
- `chunk_markdown.py` — splits `normalized/**/*.md` by H2 headings into `<pack>_chunks.generated.jsonl`. Stdlib only. **Not using LangChain by design**: its two-pass algorithm (split H2 → split oversized H2s by H3 → merge small consecutive sections back up to a size target) has no equivalent among LangChain's text splitters — reimplementing on top of them would require writing the same custom merge logic with no benefit. Run as `python rag/scripts/chunk_markdown.py <pack>`.
- `embed_and_ingest.py` — embeds every pack's `chunks/*_chunks.jsonl` with `langchain_ollama.OllamaEmbeddings` (`nomic-embed-text` via Ollama) and loads them into a `langchain_chroma.Chroma` vectorstore (backed by a `chromadb.PersistentClient` at `rag/chroma_db`, collection `ncnr_rag`). Reaches into the wrapped raw collection (`vectorstore._collection.add(...)`) to embed an enriched string (doc_id/section prefix + `search_document:` instruction prefix) while storing the original chunk text — `Chroma.add_texts()` has no precomputed-embeddings parameter. Also writes `chunk_id` into metadata so eval scripts can scope results back to a pack's own chunks (LangChain `Document` objects from similarity search don't reliably expose the Chroma-assigned id).
- `validate_pack.py` — validates a pack's required files/dirs, JSONL syntax, chunk/metadata field completeness, and cross-references chunk `source_id`s against `source_inventory.csv`. Exits non-zero on any error, which causes `run_pipeline.py` to abort. Run as `python rag/scripts/validate_pack.py <pack>`.
- `gen_chunks.py` — retrieval-only script (no LLM call): retrieves from the `ncnr_rag` Chroma collection via `similarity_search_with_score()` (filtered by `status=current`, `access_level` cascade, optional `instrument`/pack; score is cosine distance) and prints the top-k matching chunks. Its `retrieve()`/`format_chunks()`/`RetrievalError` are imported and reused in-process by `mcpServer.py`'s `gen_chunks` tool. `--max-distance D` (default `0.5`, range 0–2) drops chunks beyond D. Run as `python rag/scripts/gen_chunks.py "<question>" [--pack PACK] [--top N] [--max-distance D] [--access-level public|internal|restricted]`.
- `test_retrieval_embedding.py` — embedding-based retrieval evaluation: runs each pack's `eval/*.jsonl` questions against the `ncnr_rag` Chroma collection and reports top-1/top-k accuracy and MRR. Pack-scoped chunk filtering uses the `chunk_id` metadata field written by `embed_and_ingest.py`. Run as `python rag/scripts/test_retrieval_embedding.py <pack> --evaluate [--detail]` or `--evaluate-all [--output-csv file.csv]`.
- `evaluate_retrieval_ragas.py` — embedding-based retrieval evaluation using RAGAS-standard Context Precision@K and Context Recall, computed directly from each eval question's `expected_sources` ground truth. Retrieves the same way as `test_retrieval_embedding.py`. Run as `python rag/scripts/evaluate_retrieval_ragas.py <pack> [--top N] [--detail]` or `--evaluate-all [--output-csv file.csv]`.
- `_common.py` — shared helpers imported by the other scripts: pack list (`PACKS`), Chroma bootstrap (`open_vectorstore`), JSONL/eval loading, Ollama health-check/auto-start (`ensure_ollama`), and the eval CSV writer. chromadb/langchain are imported lazily inside `open_vectorstore()` so stdlib-only callers stay light.

## Agent & serving layer (`scripts/`)

- `mcpServer.py` — **FastMCP server** (`python scripts/mcpServer.py`, stdio transport) exposing fifteen tools. `gen_chunks` runs retrieval in-process (reusing a single cached Chroma handle, opened once at startup) rather than as a per-call subprocess. The tools:
  - `run_pipeline` — runs the full ingestion pipeline as a subprocess.
  - `gen_chunks` — semantic retrieval from the `ncnr_rag` Chroma vectorstore; wraps results in `<retrieved_chunks>` tags.
  - `generate_plot` — `exec`s model-authored Plotly code (with `go`/`px`/`np` pre-imported; figure assigned to `fig`), serializes the figure JSON under `static/generated/`, and returns a `<div class="plotly-figure" data-src=...>` placeholder the UI renders. For **ad-hoc/user-supplied** x/y data only — not reduced curves (see `plot_reduction`).
  - `plot_reduction` — reduces files exactly as `reduce_files` (same instrument/template/node_files/target_node), then fetches the target node's `return_type="plottable"` output and builds the Plotly figure server-side from the **full** Q/intensity arrays (via `_plottable_traces`, which reads reductus' type-`nd` plottable: `options.xcol/ycol` → `datas[col]["values"]` + errorbars + axis labels). Returns the same `<div class="plotly-figure">` placeholder as `generate_plot`. This is the correct way to plot a reduced curve: `reduce_files` truncates arrays to an ~8-sample summary to fit the context window, so plotting its output via `generate_plot` yields an empty chart.
  - `list_instruments`, `get_instrument`, `list_datasources`, `list_data_files` — thin wrappers over `reductus.web_gui.api`.
  - `find_raw_data_paths` — looks up an NCNR experiment's raw files by experiment_id/instrument via the NCNR metadata API (`/datafiles`), attaches each file's real mtime (required by reductus) and a `download_url` (served by `app.py`'s `/download/raw` proxy), plus a best-effort `intent` read straight from the metadata DB.
  - `list_reduction_templates` — lists an instrument's reduction templates and their file-input nodes/intents.
  - `reduce_files` — runs selected files through a named reduction template without hand-building the module graph. Computes each node's output via `calc_terminal` (working around a latent `calc_template` unpacking bug); results are compacted/truncated to a fixed size budget (`_fit_result`) so they fit an LLM context window.
  - `export_reduction` — reduces exactly as `reduce_files` and writes the target node's output to a downloadable ORSO file (`orso_text` → `.ort`, `orso_nexus` → `.orb`) under `static/generated/exports/`, served by `app.py`'s `/download/export` endpoint.
  - `get_file_intent` — determines a single raw file's measurement intent. Fast path reads the harvested intent from the NCNR metadata DB by filename (no file load); falls back to loading just the file header through reductus (resetting the load node's baked-in intent to `auto`) when the DB has none.
  - `inspect_raw_file` — opens a raw NeXus/HDF5 file and reads the free-text descriptions, run comments/annotations, sample info, and any logged field that live *inside* the file — richer than the harvested metadata DB that `find_raw_data_paths`/`get_file_intent` see. Fetches the bytes with `reductus.dataflow.fetch.url_get` (cached HTTP GET from `https://ncnr.nist.gov/pub/`, same `{path, mtime, source}` descriptor `find_raw_data_paths` emits) and opens them via `reductus.dataflow.lib.h5_open.h5_open_zip` (handles NCNR's zipped `.nxz`/`.nxs.cdr` variants). Returns one dict per NeXus entry: a curated `description` (hardcoded NeXus/NCNR paths present in the file, plus a value-deduped keyword scan for free-text fields), a breadth-first `structure` outline (shape+dtype only, node/depth-capped), and `requested_fields` for any explicit NeXus paths passed in `fields`. Response is size-budgeted via `_fit_result`.
  - `find_experiment_logsheet` — searches the NG7 / NGB30 experiment log-sheet PDF archives and returns download links so the agent can hand a matching PDF to the user. Reads charlotte's **public HTTP mirror** of the SANS data share (Apache auto-index; base URLs in `LOGSHEET_ROOTS`, e.g. `https://charlotte.ncnr.nist.gov/pub/sansdata/NG7_FILES/Experiment%20Log%20Sheets/`) rather than the `\\charlotte.ncnr.nist.gov\Sans Data\...` UNC roots it used to walk — the UNC paths only resolve on a NIST-networked host with the share mounted, the URLs work anywhere. `_walk_logsheet_index` crawls the index recursively (`_INDEX_ROW` parses one autoindex row: href + last-modified + human size; `html.unescape` the href, since Apache HTML-escapes `&` rather than percent-encoding it), skipping macOS `._*` AppleDouble stubs, capped at `max_dirs=300` (the real trees are 81/44 dirs) and cached for `_LOGSHEET_INDEX_TTL` (900s) since a full walk is ~50 requests/~3s. `instrument` accepts `ng7`/`ngb30` (case/punctuation-insensitive via `_normalize_instrument_token`); `query` matches all whitespace-separated terms case-insensitively against each PDF's relative path, sorted most-recent-first. `size_kb` now comes from the index's rounded size cell (`_autoindex_size_kb`), so it's approximate and may be null — exact bytes would cost a HEAD per file. Files are named by date + PI surname (`YYYYMMDD_<instr>_<PI>.pdf`), **not** by experiment/proposal number, so search by PI/date. Returns a **dict** `{"instrument", "query", "count", "matches"}` (each match `{filename, relative_path, size_kb, modified, download_url}`), served by `app.py`'s `/download/logsheet` endpoint. Returns a dict rather than a bare list on purpose: langchain-core treats a tool's returned list as message-content blocks when every element qualifies (true for an empty no-match list), passing it to the model provider un-stringified, which the OpenAI/RChat API then rejects (`content.0 is not a valid Content`); a top-level dict is always JSON-stringified.
- `app.py` — **FastAPI web-UI server** (`python scripts/app.py`, http://127.0.0.1:8000). Serves `static/index.html` and a `/chat` + `/chat/stream` (SSE) API, plus `/download/raw`, `/download/export`, and `/download/logsheet` (reads `mcpServer.LOGSHEET_ROOTS`, proxies the log-sheet PDF from charlotte's public mirror and streams it inline so it opens in the browser's PDF viewer; path-traversal guarded, and the fixed roots keep it from becoming an open SSRF relay) proxies and `/api/models` / `/api/key-status`. Bring-your-own-key: `_provider_for_model` routes a chosen model to OpenAI/Anthropic/Google/RChat, and `_build_agent` constructs a per-request LangGraph `create_agent` from that provider's key. Key behaviors:
  - Loads `mcpServer.py`'s tool functions in-process (wrapped by `_safe_tool` so a single tool exception is returned as an error string instead of aborting the run) **plus** OpenAPI tools from `@ivotoby/openapi-mcp-server` over the NCNR metadata API (`openAPI.json`).
  - **Per-request tool scoping** (`_scoped_tools`/`_select_tool_names`): groups the tools by workflow (knowledge_base / search / reduction / plot / admin) and binds only the groups a message's keywords implicate (plus their dependency groups), falling back to the full set when intent is ambiguous — this trims the fixed input-token cost of re-sending every tool's schema on every model step. Unrecognized (newly added) tools are always kept.
  - **Context-editing middleware** (`ContextEditingMiddleware` + `ClearToolUsesEdit`) clears old tool outputs from what's sent to the model once the running token total gets large, without touching the checkpointer's record.
  - `ReasoningChatOpenAI` subclass preserves the `reasoning_content` streaming delta (gpt-oss/DeepSeek-style reasoning that base ChatOpenAI drops) so the UI can show live "thinking"; `_extract_reasoning` also handles Anthropic extended-thinking blocks.
  - `AGENT_RECURSION_LIMIT = 100` (LangGraph default 25 cut off long multi-item runs); the MULTI-ITEM system prompt tells the model to fan per-item tool calls out in parallel in a single turn.
  - `_TurnMetrics` logs a per-turn breakdown (model calls, ttft, wall time, token counts, per-tool durations) to the `ncnr.timing` logger and streams a `metrics` event to the UI.
- `agent.py` — CLI REPL agent, fixed to the NIST RChat `gpt-oss-120b` model (set `RCHAT_API_KEY` in `.env`). Shares the same MCP tool set and system prompt shape as `app.py` but with a single static model and no BYOK/scoping. Run as `python scripts/agent.py`.

> **Adding a new `@mcp.tool`:** neither front-end auto-discovers tools. Each builds its LangChain tool set from a hand-maintained `MCP_TOOL_NAMES` allowlist (`getattr(mcpServer, name)`), so a new tool must be added to `MCP_TOOL_NAMES` in **both** `app.py` and `agent.py` to be usable — the two lists are maintained separately and are not identical (e.g. `agent.py` omits `export_reduction`). For `app.py`, also add the tool to the right `TOOL_GROUPS` bucket (and, if it needs new trigger words, `GROUP_SIGNALS`) so per-request scoping selects it; an unlisted-but-loaded tool is never dropped, but it also won't pull in its workflow group. A `TOOL_LABELS` entry gives it a friendly streaming status. Registering it with `@mcp.tool()` alone only exposes it over the stdio transport, not to the in-process agents. **Return a `dict`/`str`, not a bare `list`:** langchain-core's `_format_output` keeps a tool's returned list as message-content blocks (instead of JSON-stringifying it) whenever every element looks like a content block — which is vacuously true for an empty list and true for a list of strings — and that list is then sent to the OpenAI/RChat API un-stringified, which rejects it (`content.0 is not a valid dictionary or instance of Content`). Wrap list results in a dict (e.g. `{"count", "matches"}`) so they always stringify.
- `test_reductus_tools.py` — smoke-tests the reductus tools directly (no MCP transport) against the real reductus API and live NCNR servers. Run as `python scripts/test_reductus_tools.py` (needs network; importing `mcpServer.py` also starts Ollama if it isn't running).
- `test_get_file_intent.py` — tests `get_file_intent` accuracy against real NCNR reflectometry/CANDOR data using the metadata API's trajectory-intent ground truth (plus the two error paths). Run as `python scripts/test_get_file_intent.py` (needs network).

> **Model note:** the RChat-hosted `gemma-4-31B-it` model cannot disambiguate between tools under `tool_choice="auto"` once 2+ tools are bound (it returns a blank tool call with no name/id, which crashes `ToolMessage` construction). `gpt-oss-120b` and `NVIDIA-Nemotron-3-Super-120B-A12B-FP8` handle multi-tool selection correctly; `agent.py` uses `gpt-oss-120b`.

## requirements.txt

Pins `chromadb`, `pypdf`, `requests`, `openai`, the LangChain packages (`langchain`, `langchain-core`, `langchain-ollama`, `langchain-chroma`, `langchain-openai`, `langchain-anthropic`, `langchain-google-genai` — the slim per-integration packages), and the agent-layer packages (`fastmcp`, `reductus`, `plotly`, `numpy`, `langgraph`, `langchain-mcp-adapters`, `python-dotenv`, `fastapi`, `uvicorn`). None pull in `sentence_transformers`/PyTorch.
