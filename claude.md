# rag-knowledge-pack-template

A RAG knowledge-pack template for NIST NCNR neutron-scattering instrument documentation. Four packs: `candor/`, `vsans/`, `nse/` (instrument-specific) and `common/` (shared NICE/NCNR-wide docs).

## Pack structure (per `PACK_STRUCTURE.md`)

Each pack (`<pack>/`) contains:
- `README.md`, `source_inventory.csv`, `manifest.jsonl`, `glossary.yaml`, `access_policy.yaml`, `ingest_config.yaml`
- `originals/` — unmodified source material (web_pages, pdfs, papers, manuals, notebooks, scripts, data_examples)
- `normalized/` — Markdown docs with YAML frontmatter (doc_id, source_id, instrument, workflow_stage, source_type, access_level, status, owner, last_reviewed, source_url_or_path, citation_required), organized into stage subfolders (overview, instrument_control, experiment_planning, data_access/raw_data, metadata, sample_environment, troubleshooting, citations_publications, etc.)
- `chunks/` — JSONL chunk records (chunk_id, doc_id, source_id, text, metadata) generated from `normalized/`
- `eval/` — JSONL evaluation questions for retrieval testing
- `review/` — review artifacts

Schemas for these record types live in `schemas/`.

## Scripts (`scripts/`)

The embedding/vectorstore/chat-model layer is built on LangChain (`langchain-core`, `langchain-ollama`, `langchain-chroma`, `langchain-openai`) rather than raw HTTP/`chromadb` calls — see the per-script notes below for what each LangChain abstraction replaces.

- `chunk_markdown.py` — splits `normalized/**/*.md` by H2 headings into `<pack>_chunks.generated.jsonl`. Stdlib only. **Not migrated to LangChain by design**: its two-pass algorithm (split H2 → split oversized H2s by H3 → merge small consecutive sections back up to a size target) has no equivalent among LangChain's text splitters (`MarkdownHeaderTextSplitter` only splits, never merges back up; `RecursiveCharacterTextSplitter` has no heading awareness) — reimplementing on top of them would just mean writing the same custom merge logic underneath, for no benefit.
- `embed_and_ingest.py` — embeds every pack's `chunks/*_chunks.jsonl` with `langchain_ollama.OllamaEmbeddings` (`nomic-embed-text` via Ollama) and loads them into a `langchain_chroma.Chroma` vectorstore (backed by a `chromadb.PersistentClient` at `./chroma_db`, collection `ncnr_rag`). Because `Chroma.add_texts()`/`.from_documents()` always re-embeds `page_content` themselves (no precomputed-embeddings parameter) and this pipeline needs to embed an *enriched* string (doc_id/section prefix + the `search_document: ` instruction prefix) while storing/displaying the *original* chunk text, ingestion reaches into the wrapped raw collection (`vectorstore._collection.add(...)`) instead of going through `add_texts()`. Also writes `chunk_id` into each chunk's metadata (in addition to the existing `doc_id`/`source_id`) — needed because LangChain `Document` objects returned from Chroma similarity search don't reliably expose the Chroma-assigned id, so callers that need to scope results back to a pack's own chunks (the eval scripts below) read it from metadata instead. Requires a running Ollama instance with `nomic-embed-text` pulled (`ollama pull nomic-embed-text`).
- `test_retrieval.py` — stdlib-only TF-IDF retrieval/evaluation tool. **Not migrated to LangChain by design**: it's a deliberate, dependency-free baseline that never touches Chroma/Ollama/embeddings at all, used as a fixed reference point the embedding-based scripts are diffed against — changing its implementation would invalidate that comparison.
  - `python scripts/test_retrieval.py <pack> "<query>" [--top N]` — single query
  - `python scripts/test_retrieval.py <pack> --evaluate [--detail]` — evaluate one pack against its eval questions
  - `python scripts/test_retrieval.py --evaluate-all [--output-csv file.csv]` — evaluate all packs, writes `retrieval_results.csv`
  - Loads ALL `*.jsonl` files under a pack's `chunks/` dir, so a pack must not contain duplicate/derived chunk files alongside the canonical one.
- `validate_pack.py` — validates a pack's required files/dirs, JSONL syntax, chunk/metadata field completeness, and cross-references chunk `source_id`s against `source_inventory.csv`. Run as `python scripts/validate_pack.py <pack>`. Purely a structural/schema linter over on-disk files — unaffected by the LangChain migration.
- `query_rag.py` — end-to-end RAG query against an LLM. Retrieves from the `langchain_chroma.Chroma` `ncnr_rag` collection via `similarity_search_with_score()` (filtered by `status=current`, `access_level` cascade, optional `instrument`/pack; the raw score surfaced is cosine *distance*, used by `--max-distance` below — no cross-encoder reranking, relies on Chroma's vector-search ranking directly), then runs an LCEL chain (`ChatPromptTemplate | llm | StrOutputParser()`) with the retrieved chunks as context. Three backends, all exposed as a `langchain_core.BaseChatModel` so the chain is backend-agnostic: `--backend ollama` (default) uses `langchain_ollama.ChatOllama` against a local Ollama server (`ollama serve`); `--backend openai` uses `langchain_openai.ChatOpenAI` against any OpenAI-compatible chat-completions endpoint (e.g. an NVIDIA NIM container, such as `nvidia/llama-3.3-nemotron-super-49b-v1.5`) reached via a manually-established SSH tunnel (`ssh -L <port>:localhost:<port> user@host`) — requires `--base-url` and `--model`, with optional `--api-key`/`NIM_API_KEY` env var (falls back to a placeholder key if none given, since `ChatOpenAI` requires a non-empty string even against unauthenticated endpoints); `--backend ssh` uses a custom `SSHChatModel` (a small `BaseChatModel` subclass wrapping the same paramiko/curl logic as before — there's no existing LangChain integration for "SSH exec a curl against a remote loopback") to SSH into the remote host itself and curl the NIM container's own loopback directly for chat — no tunnel needed — requires `--model` plus `--ssh-host`/`--ssh-user` and `--ssh-password` or `--ssh-key-file` (or the `GB10_SSH_HOST`/`GB10_SSH_USER`/`GB10_SSH_PASSWORD`/`GB10_SSH_KEY_FILE` env vars). Embedding always goes through local Ollama (`nomic-embed-text`) regardless of chat backend. `--max-distance D` (default `0.5`) skips the LLM call entirely if the closest retrieved chunk's cosine distance exceeds `D` (range 0–2, lower is stricter; pass `2` to disable). Requires `embed_and_ingest.py` to have been run first, a running Ollama instance with `nomic-embed-text` pulled (and a chat model pulled for `--backend ollama`, e.g. `ollama pull llama3.2`), plus `paramiko` for `--backend ssh`. Run as `python scripts/query_rag.py "<question>" [--pack candor] [--top N] [--backend ollama|openai|ssh] [--base-url URL] [--model NAME] [--api-key KEY] [--ssh-host HOST] [--ssh-user USER] [--max-distance D] [--access-level public|internal|restricted]`.
- `test_retrieval_embedding.py` — embedding-based counterpart to `test_retrieval.py`'s TF-IDF evaluation: runs each pack's `eval/*.jsonl` questions against the `langchain_chroma.Chroma` `ncnr_rag` collection (via `embed_and_ingest.py` output, embedding queries through `langchain_ollama.OllamaEmbeddings`) and reports top-1/top-k accuracy and MRR, for apples-to-apples comparison with the TF-IDF baseline. Pack-scoped chunk filtering reads the `chunk_id` metadata field written by `embed_and_ingest.py`. Requires a running Ollama instance with `nomic-embed-text` pulled. Run as `python scripts/test_retrieval_embedding.py <pack> --evaluate [--detail]` or `--evaluate-all [--output-csv file.csv]`.
- `evaluate_retrieval_ragas.py` — embedding-based retrieval evaluation using RAGAS-standard Context Precision@K and Context Recall, computed directly from each eval question's `expected_sources` ground truth (no fuzzy text-similarity approximation — this custom metric math is intentionally untouched by the LangChain migration). Retrieves the same way as `test_retrieval_embedding.py` (`langchain_chroma.Chroma` + `langchain_ollama.OllamaEmbeddings`, scoped by the `chunk_id` metadata field). Requires `embed_and_ingest.py` to have been run first, plus a running Ollama instance with `nomic-embed-text` pulled. Run as `python scripts/evaluate_retrieval_ragas.py <pack> [--top N] [--detail]` or `--evaluate-all [--output-csv file.csv]`.

`requirements.txt` pins `chromadb`, `paramiko`, and the LangChain integration packages (`langchain-core`, `langchain-ollama`, `langchain-chroma`, `langchain-openai` — the slim per-integration packages, not the umbrella `langchain` package, since this repo only needs embeddings/vectorstore/chat-models/LCEL, not agents/memory/legacy chains). None of these pull in `sentence_transformers`/PyTorch, so the ~600MB torch dependency is still avoided. All embedding-dependent scripts require Ollama running locally with `nomic-embed-text` pulled (`ollama pull nomic-embed-text`); `query_rag.py`'s default `--backend ollama` additionally needs a chat model pulled (e.g. `ollama pull llama3.2`).
